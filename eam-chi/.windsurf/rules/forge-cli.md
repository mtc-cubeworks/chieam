# Forge CLI — Entity & Migration Toolkit

**Location:** `backend/app/forge.py`

## Purpose

Forge is the **single CLI** for entity scaffolding, model generation, and database migrations. It replaces manual Alembic commands with a fast, safe, and consistent workflow.

---

## File Triggers — When to Use Forge

| You Changed                 | Forge Command                            | What It Does                                            |
| --------------------------- | ---------------------------------------- | ------------------------------------------------------- |
| **Entity JSON** (`*.json`)  | `forge sync`                             | Regenerates model → checks DB → auto-migrates if needed |
| **Added field to JSON**     | `forge sync -m "add field"`              | Updates model → generates migration → applies           |
| **Removed field from JSON** | `forge sync -m "remove field"`           | Updates model → generates migration → applies           |
| **Changed field type**      | `forge sync -m "change type"`            | Updates model → generates migration → applies           |
| **Model file directly**     | `forge sync`                             | Detects change → generates migration → applies          |
| **Need new entity**         | `forge new-entity <name> --module <mod>` | Creates JSON + model + optionally migrates              |
| **Need to rollback**        | `forge migrate --rollback [N]`           | Rolls back N migrations (default 1)                     |

**Key insight:** `forge sync` is smart — it skips DB round-trips when nothing changed (~0.0s), and only runs migrations when models actually changed (~3-5s).

## Core Commands

### Everyday Workflow

```bash
# Push everything to DB (fast path if no changes)
forge sync

# Push with custom migration label
forge sync -m "add location field"
```

**Performance:**

- Zero changes: **~0.0s** (skips Alembic entirely)
- Model changes: **~3-5s** (generates + applies migration)

### Migration Control

```bash
# Check current DB revision + history
forge migrate --status

# Apply pending migrations only (no autogenerate)
forge migrate --apply-only

# Rollback last migration
forge migrate --rollback

# Rollback 3 migrations
forge migrate --rollback 3

# Generate + apply in one step (manual workflow)
forge migrate -m "custom migration"
```

### Entity Scaffolding

```bash
# Create new entity with JSON + model
forge new-entity report --module reporting

# Create with initial fields
forge new-entity spare_part --module purchasing_stores \
  --fields "name:string,item:link,qty:int"

# Add field to existing entity (updates JSON only)
forge add-field purchase_request priority string

# Regenerate model from JSON (one entity)
forge update-model purchase_request

# Regenerate all models (force)
forge update-model --all
```

### Utilities

```bash
# Show DB revision
forge status

# Interactive Python REPL with app context
forge console

# Run seed_data()
forge seed
```

## Workflow Patterns

### Pattern 1: Edit JSON in Model Editor → Sync

```bash
# 1. Edit entity JSON in Model Editor UI
# 2. Run sync
forge sync -m "add cost_code field"

# Done — model regenerated, migration applied
```

### Pattern 2: Add Field via CLI

```bash
# Add field to JSON
forge add-field employee phone string

# Sync to DB
forge sync -m "add phone to employee"
```

### Pattern 3: Manual Model Edit → Sync

```bash
# 1. Edit model .py file directly
# 2. Run sync
forge sync -m "custom model change"

# Forge detects model change → generates migration → applies
```

### Pattern 4: Rollback + Re-apply

```bash
# Rollback last migration
forge migrate --rollback

# Fix issue, then re-apply
forge migrate --apply-only
```

## Safety Guards

### Reserved Tables (Cannot Generate)

Forge **refuses** to generate entity models for these tables (owned by `app/models/*`):

```
users, roles, entity_permissions, user_roles,
workflow_states, workflow_actions, workflows,
workflow_state_links, workflow_transitions,
module_orders, entity_orders, attachment, audit_log
```

Attempting to create an entity with these table names will fail with an error.

### BaseModel Fields (Auto-Skipped)

Forge **never** generates these fields in entity models (inherited from `BaseModel`):

```
id, created_at, updated_at
```

If your JSON includes these fields, they're silently skipped during model generation.

## File Structure Reference

```
app/modules/{module}/
├── entities/
│   └── {entity}.json          ← Edit this
├── models/
│   ├── __init__.py            ← Auto-updated by Forge
│   └── {entity}.py            ← Auto-generated by Forge
└── apis/
    └── {entity}.py            ← Business logic (manual)

alembic/versions/
└── {timestamp}_{label}.py     ← Generated by Forge
```

**Trigger chain:**

1. Edit `{entity}.json` → `forge sync`
2. Forge regenerates `{entity}.py` (if JSON newer than model)
3. Forge runs `alembic check` (if any model changed)
4. Forge generates + applies migration (if DB out of sync)

## Production Workflow

### Development

```bash
# Edit entity JSON
# Run sync
forge sync -m "descriptive label"

# Commit both JSON and migration file
git add app/modules/*/entities/*.json
git add alembic/versions/*.py
git commit -m "Add field X to entity Y"
```

### CI/CD Deploy

```bash
# In deployment script — only apply, never autogenerate
forge migrate --apply-only
```

**Never** run `forge sync` or `forge migrate` (autogenerate) in production. Always generate migrations in dev and commit them.

## Common Issues

### "Table 'X' is already defined"

**Cause:** Duplicate entity JSON creating a model that conflicts with `app/models/*`.

**Fix:** Delete the conflicting entity JSON. Reserved tables are protected by Forge guards.

### "No changes detected — skipping DB check"

**Cause:** No model files changed since last sync.

**Fix:** This is normal (fast path). If you expect changes, verify your JSON edits were saved.

### Migration takes 3+ minutes

**Cause:** Remote dev DB with high network latency.

**Fix:** Use local DB for dev, or accept the latency. Forge skips DB entirely when no changes detected.

## Architecture Notes

- **Progress bar:** Shows entity scan → migration generation → application
- **Fast path:** Skips Alembic subprocess when no models changed (pure Python check)
- **Error handling:** `alembic/env.py` wraps each model import in try/except — one broken model won't crash migrations
- **Import order:** `app/models/*` loaded first, then `app/modules/*/models/*` to prevent table conflicts
